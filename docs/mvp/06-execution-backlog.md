# Execution Backlog

## Purpose

Convert the MVP documentation into follow-up tasks for product, content, design, prototype engineering, and validation.

This backlog does not implement code. It defines what to do next.

## Product Tasks

### P1: Lock MVP Name Direction

Deliverable:

- 10 name candidates.
- 3 positioning taglines.
- Recommendation for US-first test name.

Acceptance:

- Name communicates signal detection or relationship strategy in under 3 seconds.

### P2: Define MVP Event Taxonomy

Deliverable:

- Analytics event list for case start, tag select, response select, report view, persona select, growth upgrade, investment decision.

Acceptance:

- Each validation metric in `05-validation-plan.md` maps to at least one event.

### P3: Monetization Hypothesis

Deliverable:

- Hypothesis doc for hints, rewinds, premium cases, persona skins, and report card cosmetics.

Acceptance:

- No monetization blocks the base case loop.

## Content Tasks

### C1: Write 10 MVP Cases

Deliverable:

- 10 cases using the content bible grammar.

Acceptance:

- Each case has 5-8 evidence units, 2-4 correct tags, 3-4 response options, and one result report.

### C2: Expand Male Archetype Sheets

Deliverable:

- 5 archetype sheets with surface charm, hidden risk, green flags, red flags, and sample lines.

Acceptance:

- Each archetype can support at least 2 cases.

### C3: Write Result Report Line Bank

Deliverable:

- 50 short report insights.

Acceptance:

- Lines are screenshot-friendly and avoid gender-war universals.

### C4: Create Investment Node Scripts

Deliverable:

- 3 Add/Hold/Fold/Test nodes with outcomes.

Acceptance:

- Nodes read as relationship/emotional investment, not financial trading or manipulation advice.

## Design Tasks

### D1: Low-Fidelity iOS Flow

Deliverable:

- iPhone-sized wireflow for Cases, Persona Select, Chat, Tagging, Response, Result, Growth, Portfolio.

Acceptance:

- A full case is playable through wireframes.

### D2: Result Report Card Layout

Deliverable:

- 3 share-card layout options.

Acceptance:

- Card works in one mobile screenshot.

### D3: Persona Card Visual System

Deliverable:

- 4 persona card layouts.

Acceptance:

- Cards communicate style and gameplay effect without long text.

### D4: Visual Direction Board

Deliverable:

- 2-3 style directions for US-first, Japan-compatible female casual positioning.

Acceptance:

- Direction avoids generic otome, generic beige lifestyle, and male-gaze dress-up.

## Prototype Engineering Tasks

### E1: Choose Prototype Path

Options:

- Clickable Figma prototype.
- Mobile web prototype with iOS viewport.
- Native iOS prototype.

Recommendation:

- Start with clickable or mobile web prototype until case desire is proven.

Acceptance:

- Chosen path supports fast content iteration.

### E2: Data-Driven Case Renderer

Deliverable:

- Schema for case, messages, clues, tags, responses, reports, rewards.

Acceptance:

- New cases can be added without UI code changes.

### E3: Basic State Model

Deliverable:

- Player stats, persona choice, case result, portfolio state.

Acceptance:

- Golden path can be completed end to end.

## Validation Tasks

### V1: Recruit Testers

Deliverable:

- 8-12 content-only testers for first round.

Acceptance:

- Participants include target-adjacent women who consume relationship or romance content.

### V2: Run Case Desire Test

Deliverable:

- Test notes for 3 cases.

Acceptance:

- Capture what players argue about, not just whether they like it.

### V3: Synthesize Learnings

Deliverable:

- One-page validation memo.

Acceptance:

- Memo recommends continue, revise, or pivot with evidence.

## Suggested Next Sequence

1. Write 10 cases.
2. Design low-fidelity iOS wireflow.
3. Build content-only test deck or mobile web prototype.
4. Run 8-12 tester sessions.
5. Decide whether to build first playable iOS prototype.

## Out Of Scope For Next Step

- Full character art pipeline.
- Full monetization implementation.
- Full localization.
- Heavy wardrobe system.
- Long-form romance routes.
- Backend accounts or social graph.
