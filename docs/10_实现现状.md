## 2. 已实现

### 玩法端（两端镜像的规则）
- 交换相邻两格的 3 连消除、下落、**随机补充**、连锁、计分（连锁递增）、步数、得分目标、无合法消除自动洗牌。
- **多连特效**（仅 Godot，C++ 不做，见 §3）：4连直线/5连彩球/T-L爆炸 + 触发链 + 彩球交换引爆。
- **A 关卡模型**（两端）：异形棋盘（WALL）/ 4 类目标（SCORE/COLLECT/CLEAR_JELLY/CLEAR_BLOCKER）/ 果冻底层 / 冰锁涂层。

### 算法端（C++ 独有）
- 裸 Core 模拟器（镜像玩法规则，**无特效**）。
- 求解器组：`random`(地板) / `greedy`(刷分) / 画像面板 `rusher·scorer·cascader`(目标感知天花板)。
- 评估：求解器组多次投票 → 难度档 + LFHC 差距 + 通过率 + 节奏分。
- 生成三法：`generate_and_test`（造+筛）/ `generate_for_difficulty`（二分目标命中难度带）/ `generate_fi2pop`（双种群 GA）。

## 3. ★与设计（09/05）的关键修正★（实现里学到的）

1. **关卡≠确定序列**（已并入 04/08/09）：消除是带随机补充的序列决策，关卡=棋盘+目标+步数+**补充随机分布**，无"出现序列"。难度/最优解都是**期望**意义（多次模拟取分布）。

2. **求解器必须目标感知，不是最大化分数**：原 09 伪代码假设最大化 score。加了目标/障碍后这是错的——一个刷分求解器会把"收集/清果冻/破冰锁"关误判成过不去。改为 `move_value`：一步价值 = 朝当前目标推进多少（消目标色/清层/破层），分数降为次要。
   - **目标导向标定（已闭合）**：目标**可解性、难度评估、目标数标定**现已全部目标感知。`evaluate_level` 用 `rusher·scorer·cascader` 画像投票验收；生成器给 COLLECT/果冻关定目标数时，用 `smart_greedy` **真去追该目标**量出的天花板来标定（`generate_and_test` 直接据此定数；`generate_for_difficulty` 据此定二分搜索上界），不再用 `greedy_play` 刷分顺带量。

3. **解空间计数节奏曲线在随机补充下失效**：09 §3.6 假设"解空间随每步递减(前松后紧)"，但随机补充让盘面常满、可选有效交换数平稳 → 该曲线≈平、无区分度。**改用"能推进目标的交换数"曲线**：目标关随障碍/果冻耗尽自然递减，前松后紧才真出现（冲分关本就平）。

4. **validateSolvable 升级为目标可解**：原来只查"有没有合法移动"。目标关的"可解"=**目标感知天花板赢得了**（skilled_pass>0）。生成器据此丢弃"目标定太高、谁都赢不了"的关。

5. **障碍：冰锁=经典锁(licorice)模型**（09 未细化，实现取舍；已从早期"即破即消的涂层"升级为经典锁）：冰=锁=每格 hp。锁住的格(coat>0)**不可主动交换**（`is_legal_swap` 拒绝）、**不参与消除**（`find_matches` 视其为不可匹配、断同色串）、**重力下固定不动**（`apply_gravity` 像墙一样把列切段）；只被**内/相邻消除破锁**（每次 -1），**破锁前 tile 不被清、不下落；coat 归 0 后才变普通道具可消**（经典甘草锁手感）。彩球/特效同样不清锁住格。果冻则是独立底层（消除覆盖即清层，棋子照常）。两端逐行镜像。

6. **模拟器=裸 Core，无特效（明确决定）**：特效（4/5/T-L）是 **Core juice / 表现与策略深度**（04 §2.3 规则层的深度，**不是**角色/铭文那层的 Meta），但**不进入 C++ 裸 Core 评估**；评估只在裸 Core 上做（合 09 §0"裸 Core 为基准"）。所以 C++ 端无特效、Godot 端有（C++ 侧原先残留的 `Special` enum 已删，避免误导性接口）。
   - **代价（须知，迁移风险）**：C++ 给出的难度档（HARD/EXPERT）是**无特效版**难度；真机 Godot 有特效（抬分/助清），同一关玩家实际体验可能**更易**。难度标签未必直接迁移到有特效的真实玩法——这正是 §5.1"打通两端、真机验证"必须做的原因。

7. **species = 抽象整数 ID**（与主题解耦）：引擎只认"几号和几号相同"，与是水果/魔法道具无关。Godot 用 5 个色+形（●▲■◆✶）占位渲染，**还没接 `06` 的 20 种具体道具**。

8. **三种生成法并存**：09 列了多种生成算法，实现选了 `gen-test`（最贴合、最快铺量）为主，补 `定向难度二分` 和 `FI2Pop`。MAP-Elites / CP 约束生成暂未做（gen-test+FI2Pop 已覆盖需求）。

## 4. 两端如何保持一致

**纪律：每个玩法规则先在 C++ 定+测，再镜像到 GDScript（或反之），两端各自有测试。** 模拟器必须和真人玩的是同一个游戏——否则算法评估的难度无意义。RNG 各自 seed（不要求跨语言同序列，因补充本就随机；只要求各端内部可复现）。**因此 C++ 给的难度是"分布意义"的期望（多次模拟取均值），不对应任何单局真实 playthrough——评估靠分布、不靠单局复现。**

## 7. Meta→Core 能力接口（12 萌宠技能 → 引擎映射；设计见 `02`，实现待做）

> 把 `02` 的 12 个萌宠技能落到 Core 引擎/求解器上：每个挂哪个钩子、是否动 Core 规则、是否镜像 C++、怎么守 `01` 两条铁律。这是建 Meta 前的工程地基。

### 7.1 总原则：技能是玩家侧能力，不进 C++ 裸 Core 评估
与"特效不进模拟器"同理（§3.6）：**裸 Core 才是 `09` 难度基准**（`02` 生死线"没带萌宠也能过"）。故 12 技能**只在 Godot 端实现，不镜像 C++、不影响难度标签**——求解器永远评估"裸 Core 无技能"那个游戏。**唯一桥接：预知(#8) 反向调用求解器**（求解器玩家化），是算法↔Meta 单向出口，不改基准。

### 7.3 预知 = 运行时轻量求解器（算法↔Meta 唯一桥）
- **不跑重 C++**（合 `05`：运行时不跑重算法，C++ 只离线生成关卡库）。预知是**玩家点击时的即时查询**，用 **GDScript 端 1-ply 目标感知评分**：对每个合法交换，拷贝盘面 `resolve` 一次，按 `move_value`（消目标色/清层/破障推进 + 分数次要）排序，返回前 K 步高亮。
- K 随等级：低级 1 步、高级 3-5 步（`02` §1.1）。这正是 §3.2 `move_value` 在玩家侧的复用——求解器从"后台评难度"变"前台给提示"。
