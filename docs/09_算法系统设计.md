## 0. 三条总原则（读之前必须接受）

1. **算法默认在「裸 Core」上工作,不含 Meta。**
   - 评估关卡时,不带任何角色技能、铭文加成。算法评估的是「关卡本身的固有质量」,不被玩家的 Meta 配置污染。
   - 这对应 `01` 的生死线:「没带任何角色/铭文也能正常玩、能过」。算法在裸 Core 上保证「有解、难度合适、节奏好」,就是在守这条线。
   - Meta 对关卡的影响是**上层叠加的另一层分析**(见第 7 节),不混入裸 Core 算法。

2. **跟随行业:补充棋子是随机的。**
   - 消除后下落、顶部补充的新棋子是**实时随机生成**的(经 GitHub 调研确认,这是 match-3 行业标准做法)。
   - **后果(关键)**:关卡不是「完全信息确定性问题」,而是「带随机的序列决策」。因此——
     - 「最优解」不是一个确定值,而是**期望最优**(在随机补充下,某策略的期望表现)。
     - 求解主力是 **MCTS**(蒙特卡洛,天生处理随机),而非 A\*(A\* 只用于无随机的小规模子问题校准)。
     - 一切评估都要**多次模拟取期望/分布**,而非单次结果。

3. **求解器既是「玩家」也是「评估器」。**
   - 同一套求解器,在**运营时**代表不同水平的玩家给关卡投票(定难度);在**生成时**作为评估器衡量候选关卡(可解性/难度/节奏)。
   - 这个「生成时用求解器评估」的模式,经 GitHub 调研确认是成熟做法(见第 2.7 节 generate-and-test)。

## 1. 前提与建模（裸 Core 形式化）

### 1.1 棋盘与元素

```
Board:
  grid[H][W]            # 每格的内容
  cell ∈ {
    EMPTY,                       # 空(下落填充)
    TILE(species),               # 普通棋子,species ∈ {0..K-1}(K 种水果)
    SPECIAL(type),               # 特效棋子: LINE_H, LINE_V, BOMB(T/L), COLORBOMB(5连彩球)
    BLOCKER(type, hp),           # 障碍: ICE(冰块,hp层), LOCK(锁), JAM(果酱), ...
    WALL                         # 不可消、不可动的隔断(战场切割用)
  }
  regions[]              # 战场切割: 若干区域,区域间由 WALL 或通道连接
```

> K(水果种类数)建议小(单关同屏 4-6 种,见 `04`/`06`)。障碍种类**严格受限**(冰块/锁/果酱/底层目标等少数几种,绝不引入巧克力类「蔓延/增殖」障碍——它们会让状态空间爆炸,见 `04` 的吸收边界)。

### 1.2 状态、动作、转移

```
State:
  board                  # 当前棋盘
  moves_left             # 剩余步数
  objective_progress     # 各通关目标的完成进度
  rng_seed               # 随机数状态(补充棋子用)

Action:
  swap(a, b)             # 交换相邻两格 a,b(消除的唯一基础操作)
  # 注意: 合法 swap 必须交换后能形成消除(否则非法,除非含特效/彩球)

Transition(State, Action) -> State':       # 一步的完整后果(确定性地执行,随机只在补充处)
  1. 执行 swap
  2. resolve(board):                        # 消除-下落-补充 循环,直到稳定
       loop:
         matches = find_matches(board)       # 找所有 ≥3 连
         if matches is empty: break
         clear(matches)                      # 消除,生成特效(4连/T-L/5连)
         apply_special_effects()             # 触发特效连锁
         damage_adjacent_blockers(matches)   # 消除波及的障碍掉血/破除
         gravity(board)                      # 上方棋子下落
         refill(board)                       # ★顶部随机补充新棋子(随机性在此)
         # refill 可能产生新 matches → 循环继续(连锁)
  3. update objective_progress, moves_left -= 1
  return State'
```

**连锁(cascade)是复杂度根源**:一次 swap 可能引发「消除→下落→补充→新消除→…」的长链,且补充随机,使后果难以预测。这正是为何求解要用 MCTS(采样模拟)而非穷举。

### 1.3 通关目标（可计算的那几类，见 `04`/`08`）

```
Objective 类型(均可形式化为「进度达标」):
  SCORE(target)          # 得分 ≥ target
  COLLECT(species, n)    # 消除 n 个某种水果
  CLEAR_BLOCKER(type, n) # 清除 n 个某障碍(如冰块)——底层目标
  BOTTOM(item, n)        # 把 n 个掉落物运到底部
  LIMITED_MOVES(m)       # 上述目标在 m 步内达成(约束,非独立目标)
  MIXED(obj[])           # 多目标组合(免费的多样性)

check_win(State): 所有 objective_progress 达标 且 moves_left ≥ 0
check_lose(State): moves_left == 0 且 未达标

# 无合法消除 → 自动洗牌(reshuffle,带约束),不算输、不扣步:
function on_no_legal_move(state):
    # 当 legal_moves(board) 为空(无任何能产生消除的交换)时触发
    repeat:
        board' = shuffle_existing_tiles(state.board)   # 只重排现有棋子,不增减
    until solution_space_size(board') >= 1             # ★约束: 洗到"保证至少一个合法消除"

# → 因此"无合法消除"永远能靠洗牌继续, check_lose 只由"步数耗尽且未达标"决定
# (UI 行为见 07: 轻量洗牌动画+提示, 语气中性)
```

## 2. 生成算法（多套，逐个伪代码）

> 哲学:`01` 要求「人定规则,算法生成」「可证明、非黑箱」。以下全部是白箱方法(构造/搜索/约束求解),**不含机器学习**(EA 的 cVAE 黑箱已否决)。
> 实战中「尝试多套生成算法,各生成一大堆候选,再用求解器投票筛选」(见第 4-5 节)。本节给出候选池里的每一种。

### 2.1 构造式（Constructive）— 最快,作基线

**思路**:用规则直接「造」一个关卡,不迭代改进。快,作为大批量初始候选的来源。

```
function generate_constructive(difficulty, species_set, objective_type):
    board = empty_board(H, W)
    # 1. 填充普通棋子,保证开局无现成消除(否则开局自动连锁,体验差)
    for each cell in board (按行列序):
        repeat:
            s = random_species(species_set)
        until placing s 不会与左/上已放的形成 ≥3 连   # 经典做法(Rembound 教程)
        board[cell] = TILE(s)

    # 2. 按难度放置障碍(障碍密度由 difficulty 决定)
    n_blockers = difficulty_to_blocker_count(difficulty)
    place_blockers_with_gradient(board, n_blockers)     # 见 2.1.1 密度梯度

    # 3. 按难度切割战场(放 WALL 形成区域)
    if difficulty 高: carve_regions(board, n_regions)

    # 4. 设定目标和步数
    objectives = build_objective(objective_type, difficulty)
    move_limit = estimate_moves(difficulty)             # 粗估,后续求解校准

    return Level(board, species_set, objectives, move_limit, default_refill_dist)
```

**2.1.1 障碍密度梯度（实现「前松后紧」的优雅办法,见 `08`）**

```
function place_blockers_with_gradient(board, n):
    # 让障碍密度沿某方向(如从上到下/从中心向边缘)递增
    # → 玩家推进时,空间从开阔自然变逼仄,区域「涌现」而非硬切割
    for i in 1..n:
        cell = sample_cell_weighted_by_position(board)   # 越靠"收紧端"权重越高
        board[cell] = BLOCKER(pick_blocker_type(), hp)
```

> 构造式的产物**不保证有解、难度不精确** → 必须交给第 4 节求解器验证+评估。它的价值是「快速产出海量候选」,质量由后续筛选保证。

### 2.2 遗传算法（Genetic Algorithm）— 优化软目标

**思路**:把「一个关卡」编码成基因,用适应度函数评分,进化出高分关卡。适合优化「难度贴近目标值、节奏曲线漂亮」这类软目标。

```
# 基因编码(genotype): 一个关卡的紧凑表示
Genotype = {
  blocker_positions[],     # 障碍坐标+类型
  wall_positions[],        # 战场切割
  objective_params,        # 目标参数
  move_limit,
  refill_weights[]
}
# (初始普通棋子布局可固定为"无现成消除的随机填充",不进基因,减小搜索空间)

function genetic_generate(difficulty_target, pop_size, generations):
    pop = [random_genotype() for _ in 1..pop_size]
    for gen in 1..generations:
        # 评估每个个体的适应度
        for g in pop:
            level = decode(g)
            g.fitness = fitness(level, difficulty_target)
        # 选择 + 交叉 + 变异
        parents = tournament_select(pop)
        offspring = []
        for (p1, p2) in pair(parents):
            c = crossover(p1, p2)          # 交换部分基因(障碍/墙/参数)
            c = mutate(c)                  # 随机改动: 移动障碍/增删墙/调参
            offspring.append(c)
        pop = elitism(pop) + offspring     # 保留精英 + 新后代
    return top_k(pop)                       # 返回最优的一批关卡

function fitness(level, difficulty_target):
    # 关键: 适应度 = 关卡有多接近"我们想要的样子"
    # 用第 4 节的求解器组跑这个 level,得到指标:
    metrics = evaluate_with_solvers(level)         # 见第 4-5 节
    if not metrics.solvable: return -INF           # 无解直接淘汰(或交给 FI2Pop,见 2.3)
    score = 0
    score -= |metrics.difficulty - difficulty_target|   # 难度贴近目标
    score += rhythm_quality(metrics.solspace_curve)     # 节奏曲线漂亮(前松后紧等)
    score += lfhc_gap(metrics.vote_distribution)        # 地板低天花板高
    return score
```

### 2.3 FI2Pop（可行-不可行双种群）— 严格处理硬约束 ★关卡生成主力之一★

**思路(经 Sorensen / dungeon 论文确认)**:普通遗传算法处理「必须有解」这种硬约束很笨拙(无解个体直接丢弃,浪费搜索)。FI2Pop 维护**两个种群**:

- **可行种群**(满足所有硬约束,如「有解、无死局」):按「适应度(难度贴近、节奏好)」进化。
- **不可行种群**(违反约束):按「违反约束的程度(越少越好)」进化。
- **两群只通过「后代迁移」间接交流**:可行个体变异后若变得不可行,迁到不可行群;反之亦然。
- **价值**:不可行种群在「约束边界」自由探索,常能找到可行种群够不到的好解(最优常在约束边界上)。

```
function FI2Pop_generate(difficulty_target, generations):
    feasible   = []        # 满足硬约束的个体
    infeasible = [random_genotype() for _ in 1..N]   # 初始随机(多半不可行)

    for gen in 1..generations:
        # 1. 分类: 把每个个体按是否满足硬约束,分到两群
        all = feasible + infeasible
        feasible, infeasible = [], []
        for g in all:
            level = decode(g)
            viol = constraint_violation(level)         # 见下,=0 即可行
            if viol == 0:
                g.fitness = soft_fitness(level, difficulty_target)  # 软目标(难度/节奏/LFHC)
                feasible.append(g)
            else:
                g.fitness = -viol                       # 不可行: 适应度=负的违反度(越接近可行越好)
                infeasible.append(g)

        # 2. 各自进化(独立选择+交叉+变异)
        feasible   = evolve(feasible)        # 朝"软目标更优"进化
        infeasible = evolve(infeasible)      # 朝"违反更少"进化
        # 后代会自动因可行性变化而在下一轮被重新分类(=间接迁移)

    return top_k(feasible)                   # 只返回可行的最优关卡

function constraint_violation(level):
    # 硬约束: 用求解器组检验(见第 4 节)
    v = 0
    if not is_solvable(level):        v += W1 * 1          # 无解 → 重罚
    v += W2 * count_unreachable_cells(level)               # 战场切割导致的孤立死区
    v += W3 * objective_unreachable_amount(level)          # 目标物理上凑不够(如冰块数 < 要求)
    return v
```

> **FI2Pop 是「带硬约束的关卡生成」的标准答案**:它保证产出的关卡「一定有解、无死区、目标可达」(硬约束),同时优化「难度/节奏/LFHC」(软目标)。这是本项目生成关卡的**主力方法之一**。

### 2.4 MAP-Elites（质量-多样性）— 一次产出一批「各有特色」的好关卡

**思路**:不只要「一个最优关卡」,要「一批各具特色且都不错的关卡」(关卡库需要多样性,防腻)。MAP-Elites 在**若干特征维度**张成的网格里,每个格子保留该特征组合下的最优个体。

```
function MAP_Elites_generate(iterations):
    # 1. 定义特征维度(behavior descriptors): 你关心的"关卡长什么样"的维度
    #    例: x轴=障碍密度(低→高), y轴=战场区域数(1→5)
    #    (也可加: 主目标类型、空间松紧曲线的形状等)
    archive = empty_grid()      # 每个格子存一个 elite 个体

    # 2. 初始化: 随机生成一批,各自归入对应特征格子
    for _ in 1..init_count:
        g = random_genotype()
        place_in_archive(archive, g)

    # 3. 迭代: 选已有 elite → 变异 → 评估 → 若在其格子里更优则替换
    for _ in 1..iterations:
        parent = random_elite_from(archive)
        child  = mutate(parent)
        level  = decode(child)
        if not is_solvable(level): continue            # 硬约束(也可套 FI2Pop 思路,见下)
        feature = compute_feature(level)               # 算它落在哪个格子(障碍密度,区域数)
        quality = soft_fitness(level, target)          # 质量(难度贴近/节奏/LFHC)
        if quality > archive[feature].quality:
            archive[feature] = child                   # 该格子里更优 → 替换

    return all_elites(archive)        # 整个网格 = 一批多样且优质的关卡
```

> **MAP-Elites 与 FI2Pop 可组合**(论文里常见):用 FI2Pop 的「双种群」处理硬约束,用 MAP-Elites 的「特征网格」保证多样性。组合版称 IC MAP-Elites / Constrained MAP-Elites。
> 本项目用途:**批量生成「难度档 × 区域数 × 目标类型」各种组合的关卡**,直接填充关卡库的多样性。

### 2.5 约束编程 CP（Constraint Programming, 如 OR-Tools）— 保证硬约束的「布局」

**思路(经 PuzzleSolver / OR-Tools 调研确认)**:把「合法的关卡布局」描述成变量+约束,交给 CP 求解器(如 Google OR-Tools CP-SAT)自动解出满足所有约束的布局。**用于「布局必须满足精确结构约束」的场景**(比 GA 更精确、可证明)。

```
function CP_generate_layout(H, W, constraints):
    model = CpModel()
    # 1. 变量: 每格的内容
    cell[i][j] = model.NewIntVar(0, NUM_CELL_TYPES-1)    # 每格取值=某种内容

    # 2. 约束(声明式,求解器保证满足):
    # (a) 开局无现成消除: 任意横/竖三连不同色
    for each 三连位置 (a,b,c):
        model.Add( not (cell[a]==cell[b]==cell[c]) )
    # (b) 障碍数量约束: 冰块恰好 N 个
    model.Add( sum(cell[i][j]==ICE for all i,j) == N )
    # (c) 战场连通性: 每个非WALL区域至少有一个通道相连(防孤立死区)
    add_connectivity_constraints(model, cell)
    # (d) 目标可达性: 目标物(如冰块)数量 ≥ 通关要求
    model.Add( count(target_item) >= objective_requirement )

    # 3. 求解(可要多个解 → 多样性)
    solver = CpSolver()
    solutions = solver.SearchForAllSolutions(model, limit=M)   # 或随机采样若干解
    return [decode_to_level(s) for s in solutions]
```

> **CP 的优势**:布局的硬结构约束(数量、连通性、无开局消除)由求解器**保证满足且可证明**,这是 GA 做不到的精确性。
> **CP 的局限**:它只保证「布局满足静态约束」,**不保证「玩起来有解/难度合适」**(那要动态求解,见第 4 节)。所以 CP 常作「生成合法布局骨架」,再交给求解器评估难度。

### 2.6 SAT / ASP（布尔可满足 / 回答集编程）— CP 的近亲,更偏纯逻辑

```
# SAT(经 zebra 生成器调研确认): 把约束编码成 CNF 布尔公式,SAT 求解器(如 pycosat)求解
# ASP(经 PuzzleSolver/Clingo 调研确认): 用逻辑程序声明约束,Clingo 返回所有"回答集"=合法关卡

# 对本项目: SAT/ASP 与 CP 用途重叠(都是约束式生成布局)。
# 选型建议:
#   - 优先 CP(OR-Tools): 表达力强、有数量约束、工业级、易上手。
#   - SAT: 约束能纯布尔化时更快,但数量/算术约束要手动编码,繁琐。
#   - ASP: 表达力最强(尤其递归/可达性约束优雅),但有学习曲线;
#          论文建议"约束不需要非单调推理时,用更简单的 CP 即可"。
# 实现伪代码与 CP 同构(声明变量→声明约束→求解器返回满足解),不再重复。
```

> **本项目约束式生成的落地选择:CP(OR-Tools)为主,SAT/ASP 备选**。三者是「同一类方法的不同引擎」,接口同构(声明约束→求解),按约束复杂度选引擎。

### 2.7 generate-and-test（生成-测试）— 把「求解器」嵌进生成 ★最贴合本项目★

**思路(经 unblock-me-generator 调研确认,这是它的核心算法,且正是本项目「生成候选+求解评估+筛选」思路的现成范本)**:不追求一步生成完美关卡,而是「逐步加元素 + 每步用求解器评估 + 保留最优」。

```
# 原始范本(unblock-me-generator)的算法:
#   1. 放必需元素(目标块)
#   2. 随机放一个新块 → 解这个局面 → 记录"解的路径长度"作为该布局的价值
#   3. 重复 2 若干次(如10次),得到 N 个候选布局,各有其价值
#   4. 选价值(路径长度)最优的那个,继续加块,重复 2-4,直到达到目标难度

function generate_and_test(difficulty_target):
    level = init_level_with_objectives()        # 放目标等必需元素
    while estimated_difficulty(level) < difficulty_target:
        candidates = []
        for _ in 1..N:                          # 生成 N 个"加一个元素"的候选
            cand = level.clone()
            add_random_element(cand)            # 加一个障碍/墙/调一个参数
            metrics = evaluate_with_solvers(cand)   # ★用求解器组评估(第 4 节)
            if metrics.solvable:
                cand.value = score_toward_target(metrics, difficulty_target)
                candidates.append(cand)
        if candidates is empty: break           # 没有可行候选 → 停
        level = argmax(candidates, by=value)    # 选最优候选,继续加
    return level
```

> **为什么最贴合本项目**:它把「求解器」直接当成生成过程中的「质量传感器」,与本项目「求解器组投票评估关卡」的核心思路天然一体。`unblock-me-generator` 证明了这种「生成时反复调用求解器」的模式是可行的、成熟的。
> **代价**:每步都调求解器,慢(原范本生成一个高难关卡可达 30+ 分钟)。→ 用第 6 节的「分层筛选」和「便宜求解器粗筛」优化。

## 3. 求解算法（多套，逐个伪代码）

> 求解器有两个用途:① 运营时代表不同水平玩家给关卡投票(第 5 节)② 生成时作评估器(第 2 节)。
> 因补充随机,所有求解器都在「随机环境」下工作,核心是 **MCTS**(处理随机);其余(贪心/A*/Beam)在特定场景补充。

### 3.1 通用接口

```
# 所有求解器实现同一接口,便于"投票"时统一调用
interface Solver:
    play(level) -> Result
    # Result = { won: bool, score: int, moves_used: int,
    #            solspace_curve: int[]   # ★每一步的解空间大小(节奏数据,见3.6) }

# 因随机: 同一 solver 对同一 level 跑多次取分布
function run_solver_multi(solver, level, trials=T):
    results = [solver.play(level.with_fresh_seed()) for _ in 1..T]
    return aggregate(results)   # 通过率 = won 比例; 期望得分 = mean(score); 节奏曲线 = mean curve
```

### 3.2 合法动作枚举 + 解空间度量（所有求解器的基础）

```
function legal_moves(board):
    moves = []
    for each adjacent pair (a, b) in board:
        if is_movable(a) and is_movable(b):
            board' = swap(a, b)
            if creates_match(board', a, b) or involves_special(a, b):
                moves.append(swap(a,b))
    return moves

# ★解空间大小 = "能产生消除的有效交换数"(用于节奏曲线,见3.6)
#   注意: 不是"所有合法交换",要排除无效交换,否则高估解空间(见 08 提醒)
function solution_space_size(board):
    return count(m in legal_moves(board) where produces_clear(m))
```

### 3.3 贪心求解器（Greedy）— 模拟「不规划的普通玩家」

```
# 代表"休闲/不规划玩家": 每步只选当下最直接的收益,不考虑未来
function greedy_play(level):
    state = init(level); curve = []
    while not terminal(state):
        curve.append(solution_space_size(state.board))   # 记录节奏
        moves = legal_moves(state.board)
        if moves is empty: break
        # 贪心策略(可配多种,代表不同"普通玩家"):
        best = argmax(moves, by = lambda m: immediate_gain(state, m))
        # immediate_gain 例: 这步直接消除的棋子数 + 是否凑出特效
        state = transition(state, best)        # 含随机补充
    return Result(won=check_win(state), score=state.score,
                  moves_used=..., solspace_curve=curve)
```

> 贪心是「地板」的代言人。**关键:贪心要像真实普通玩家,不能太蠢**(见 `01` 陷阱)。可设几种贪心(消最多/优先凑特效/优先清障)代表不同普通玩家。

### 3.4 MCTS 求解器（Monte Carlo Tree Search）— 主力,代表「高水平玩家」★

**思路(经标准 MCTS 调研确认)**:四步迭代——选择(UCT)、扩展、模拟(rollout)、回传。处理随机:节点的价值是多次随机模拟的期望。

```
Node:
  state                 # 该节点的游戏状态
  parent, children
  N = 0                 # 访问次数
  W = 0                 # 累计回报(得分/是否通关)
  untried_moves         # 尚未扩展的合法动作

function mcts_play(level, time_budget):
    state = init(level); curve = []
    while not terminal(state):
        curve.append(solution_space_size(state.board))
        best_move = mcts_search(state, time_budget)    # 用 MCTS 选出当前最佳一步
        if best_move is None: break
        state = transition(state, best_move)            # 执行(含随机补充)
    return Result(won=..., score=state.score, moves_used=..., solspace_curve=curve)

function mcts_search(root_state, time_budget):
    root = Node(root_state)
    while time_remaining(time_budget):
        # 1. SELECTION: 用 UCT 从根往下走到未完全扩展的节点
        node = root
        while node.untried_moves is empty and node.children not empty:
            node = select_UCT(node)
        # 2. EXPANSION: 扩展一个未试动作
        if node.untried_moves not empty:
            m = pop_random(node.untried_moves)
            child_state = transition(node.state, m)     # ★含随机补充(每次可能不同)
            node = node.add_child(child_state, m)
        # 3. SIMULATION (rollout): 从该节点随机/启发式玩到终局
        reward = rollout(node.state)
        # 4. BACKPROPAGATION: 回传更新路径上所有节点
        while node is not None:
            node.N += 1
            node.W += reward
            node = node.parent
    return most_visited_child(root).move      # 最终选访问最多的子(最稳健)

function select_UCT(node):
    # UCT: 平衡 利用(高均值) 与 探索(少访问)
    return argmax(node.children, by = lambda c:
        (c.W / c.N) + C * sqrt(ln(node.N) / c.N))      # C = 探索常数

function rollout(state):
    # 从当前状态快速玩到终局: 用轻量策略(随机 or 简单启发,见3.5)
    s = state.clone()
    while not terminal(s):
        moves = legal_moves(s.board)
        if moves is empty: break
        m = heuristic_or_random_move(moves, s)          # 启发式 rollout 比纯随机强很多
        s = transition(s, m)                            # 含随机补充
    return evaluate_terminal(s)    # 通关=高回报; 得分归一化; 失败=低回报
```

> **MCTS 是天花板的代言人**,且天生处理随机补充(rollout 多次采样取期望)。
> **关键增强**:rollout 用「启发式」而非「纯随机」会强很多(见 3.5)——这是非黑箱地注入领域知识。
> **校准**:小规模关卡可用 A*(3.7)算精确最优,校准 MCTS 的近似程度(见 `01` 陷阱「MCTS 是近似」)。

### 3.5 启发函数（Heuristic）— 非黑箱地注入「什么局面好」

```
# 启发函数 = 把"什么样的棋盘好"形式化。用于 rollout 引导、贪心策略、Beam 评分。
# 这是"理解问题"的体现(非黑箱): 你写下你认为的好局面特征。
function heuristic_value(state):
    v = 0
    v += w1 * state.score                          # 已得分
    v += w2 * objective_progress(state)            # 目标完成度
    v += w3 * count_potential_chains(state.board)  # "一碰就连锁"的潜在布局
    v += w4 * count_specials_on_board(state.board) # 已有特效棋子(资源)
    v += w5 * board_health(state.board)            # 是否容易继续制造消除
    v -= w6 * count_hard_blockers(state.board)     # 剩余难缠障碍(越少越好)
    return v
# 不同流派玩家 = 不同权重 w[]: 激进连锁流(w3高)、清障流(w6权重高)、稳健流(均衡)…
```

> **流派画像就靠改 w[] 实现**(见第 5 节):每组权重 = 一类玩家的「价值观」。这是把「玩家类型」白箱化成算法的关键,不用任何机器学习。

### 3.6 解空间曲线（局内节奏的数据来源,见 `08`/用户需求）

```
# 每个求解器在 play 时,逐步记录 solution_space_size → 得到一条曲线
# 这条曲线 = 这一局的"节奏": 前期值大(开阔/轻松) → 后期值小(逼仄/精算) = 前松后紧
solspace_curve = [s0, s1, s2, ..., s_end]    # 每步的解空间大小

function rhythm_quality(curve):
    # 评价节奏曲线是否符合设计意图(如"前松后紧" = 单调递减且跨度大)
    # 可定义多种目标曲线形状(递减/波浪/先紧后松),按设计需要匹配
    return - distance(curve, target_shape)   # 越接近目标形状,节奏分越高
```

### 3.7 A*（精确,小规模校准用）

```
# A* 只用于"无随机的小规模子问题"(如固定一小段、或把补充临时设为确定来校准)
# 在完整随机问题上 A* 不适用(状态转移不确定)。用途: 给 MCTS 的近似程度做基准。
function a_star(start_state, goal_test):
    open = PriorityQueue()
    open.push(start_state, priority = 0 + heuristic_value(start_state))
    g = { start_state: 0 }
    visited = state_compression_set()              # 状态压缩去重(见3.8)
    while open not empty:
        cur = open.pop_min()
        if goal_test(cur): return reconstruct_path(cur)
        for m in legal_moves(cur.board):
            nxt = transition_deterministic(cur, m)  # 确定性转移(校准场景)
            key = compress(nxt)
            if key in visited: continue            # 去重
            visited.add(key)
            g[nxt] = g[cur] + 1
            open.push(nxt, priority = g[nxt] + heuristic_value(nxt))
    return no_solution
```

### 3.8 状态压缩（控制爆炸,A*/精确搜索用）

```
# 1. 紧凑编码(bitboard 思路): 每格 ceil(log2(K+blockers)) bit,整盘打包成大整数
function compress(board):
    code = 0
    for cell in board: code = (code << BITS) | encode_cell(cell)
    return code
# 2. 记忆化去重: 不同操作序列到达同一棋盘 → 只算一次(把搜索树压成 DAG) ← 最重要
# 3. 对称消除: 若规则有镜像/颜色置换对称,对称等价状态归一为一个
# 注: 连锁的全局性 → 无法干净"按列分解"(列不独立),分解式压缩受限(见早期分析)
```

### 3.9 集束搜索（Beam Search）— 介于贪心与 MCTS

```
# 每层只保留最有希望的 K 个状态往下扩展(控制宽度防爆炸)。比贪心强、比 MCTS 轻。
function beam_search(level, beam_width=K, depth=D):
    beam = [init(level)]
    for d in 1..D:
        candidates = []
        for state in beam:
            for m in legal_moves(state.board):
                candidates.append(transition(state, m))   # 含随机补充
        beam = top_K(candidates, by=heuristic_value, k=K)  # 只留最优 K 个
    return best(beam)
# 用途: 作为"中等水平玩家"画像之一(比贪心会算几步,但不如 MCTS 深)。
```

## 4. 关卡评估（求解器组「投票」）★用户核心思路★

> 评估一个关卡 = 让一组代表不同玩家的求解器各自试玩,看「几套能过、各拿多少分」(投票)。
> 因补充随机,每个求解器跑多次取「通过率 + 期望得分 + 平均节奏曲线」。

### 4.1 求解器组（选民 = 不同水平/流派的玩家）

```
SOLVER_PANEL = [
    Greedy(variant="most_clear"),       # 休闲玩家A: 每步消最多

    BeamSearch(width=3, depth=3),        # 普通玩家: 会算几步
    BeamSearch(width=8, depth=5),        # 熟练玩家: 算得更多

    MCTS(heuristic=w_steady),            # 流派: 稳健流

# 每个 solver 代表一类玩家。改 BeamSearch 深度/MCTS 算力/启发权重 = 造不同选民。

### 4.2 投票评估

```
function evaluate_with_solvers(level, trials=T):
    votes = {}
    for solver in SOLVER_PANEL:
        agg = run_solver_multi(solver, level, trials)    # 多次取期望(应对随机)
        votes[solver.id] = {
            pass_rate:   agg.won_ratio,        # 这类玩家的通过概率
            exp_score:   agg.mean_score,       # 期望得分
            curve:       agg.mean_curve        # 平均节奏曲线
        }
    return {
        solvable:           any(v.pass_rate > threshold for v in votes),   # 至少有选民能过
        vote_distribution:  votes,                                         # 完整投票分布(=人群体验地图)
        difficulty:         derive_difficulty(votes),                      # 见 4.3
        lfhc_gap:           lfhc_gap(votes),                               # 见 4.4
        solspace_curve:     representative_curve(votes)                    # 代表性节奏曲线
    }
```

### 4.3 难度（来自投票分布）

```
function derive_difficulty(votes):
    # 难度 = "有多少比例的选民能过" 的反函数
    # 几乎人人能过 → 简单; 只有顶尖几个能过 → 很难
    weighted_pass = sum(player_weight[s] * votes[s].pass_rate for s in votes)
    return difficulty_from_pass_rate(weighted_pass)   # 高通过→低难度
    # 也可输出"难度档": EASY / MEDIUM / HARD / EXPERT,按通过分布形状分档
```

### 4.4 LFHC 质量（地板低 + 天花板高）

```
function lfhc_gap(votes):
    floor_players = [Greedy*, BeamSearch(low)]   # 低水平选民
    ceil_players  = [MCTS(long), MCTS(流派)]      # 高水平选民
    floor_pass  = min pass_rate among floor_players   # 低水平能不能过(地板)
    ceil_score  = max exp_score among ceil_players     # 高水平能拿多高(天花板)
    floor_score = max exp_score among floor_players
    # 好的 LFHC: 低水平能过(floor_pass 高) 且 高水平得分远超低水平(差距大)
    if floor_pass < min_floor: return BAD          # 地板太高(新手过不去) → 砍
    gap = (ceil_score - floor_score) / floor_score
    return gap        # 越大 = 规划越有回报 = 天花板越高 = 越好
```

### 4.5 关卡质量综合判定（筛选）

```
function level_quality(level):

    if not m.solvable:               return REJECT       # 无解
    if m.lfhc_gap == BAD:            return REJECT       # 地板太高
    if m.lfhc_gap < min_gap:         return REJECT       # 没深度(怎么玩都一样)
    score = 0
    score += w_gap    * m.lfhc_gap                       # 天花板高
    score += w_rhythm * rhythm_quality(m.solspace_curve) # 节奏好(前松后紧等)
    score += w_fit    * difficulty_fit(m.difficulty)     # 难度落在目标档

```

> **"差距筛选"是本投票框架的一个特例**(地板选民 vs 天花板选民的得分差);投票框架更完整——还含中间档玩家、通过率分布、各流派表现。

### 5.3 第三层 — 难度节奏（跨关排布,张弛有度）

# 把关卡库里"已标难度"的关卡,按一条设计好的"难度起伏曲线"排成序列
function build_level_sequence(level_pool, difficulty_rhythm, player_profile):
    # difficulty_rhythm: 设计的难度起伏,如 [易,易,中,易,难(高潮),易(喘息),中,中,难...]

                             difficulty == adjust(target_diff, player_profile),  # 个性化偏移
                             rhythm_good == True,
                             objective_type != recent_types(sequence))           # 类型轮换防腻
        sequence.append(lvl)
    return sequence
```

### 5.4 个性化（基于行为,非性别 —— 见早期纠正）

```
function adjust(target_diff, player_profile):
    # player_profile 由"玩家行为"推断(非性别!):
    #   - 追求高分 vs 随便过
    #   - 规划深度(行为接近哪个 solver 画像)
    #   - 失败后是否重试
    #   - 游玩节奏
    if player_profile == CHALLENGE_SEEKER:  return harder(target_diff)   # 挑战型→整体更难
    if player_profile == RELAXER:           return easier(target_diff)   # 放松型→整体更易
    return target_diff
    # 更优: 直接给玩家"难度选择"(休闲/标准/挑战) + 行为隐式微调(见早期讨论)
```

> 三层合起来: **每步的松紧(节奏曲线)→ 每关的难易(投票难度)→ 一串关卡的起伏(难度节奏)**,且宏观层可按玩家个性化偏移。全部可计算、可调度。

### 5.5 同题竞技的 seed 机制（随机补充下的公平，解法A）

> 周赛/竞技要"所有人玩同一关比高分"。但补充是随机的(第0节),如何公平?

```
# 解法A(采用,行业标准 = 近似公平):
#   竞技关固定 rng_seed → 所有玩家的"随机补充序列"相同。
competitive_level.rng_seed = FIXED_SEED   # 同一场竞技,所有人同一 seed

# 公平性边界(必须明确,别误解):
#   ✅ 保证: "随机源相同"——补充棋子的序列对所有人一致(没有谁被特别针对)。
#   ⚠️ 不保证: "过程完全相同"——补充序列是"按操作消耗"的,
#       玩家操作不同 → 消耗时机/数量不同 → 盘面会逐渐分岔 → 严格说非绝对公平。
#   → 这是"近似公平"(统计意义),休闲消除竞技业界普遍采用(开心消消乐/Candy Crush),玩家接受。

# 若未来要"绝对公平"(可选,非首发):
#   解法B: 竞技关用 固定初始盘面 + 少步数 → 大部分分数来自初始盘面操作,
#          随机补充介入少,公平性更高。
#   解法C: 竞技关彻底确定化(无随机补充/预设不随操作变的序列)→ 绝对公平,
#          但与主玩法(随机补充)不一致,玩家会觉得竞技"不一样"。
# 首发用 A 即可。
```

## 6. 工程与成本

```
# 1. 离线生成关卡库(不实时生成): 后台跑生成+评估,产出"已评估、已标签"的关卡库。
#    上线时从库里按需取关卡(实时只做"调度/挑选",不实时跑昂贵求解)。
# 2. 分层筛选(控制求解成本): 求解器评估贵(尤其 MCTS),不能对每个垃圾候选都跑全套。
function tiered_screening(candidates):
    # 第1层(便宜): 用贪心 + 静态检查 快速粗筛,淘汰明显无解/明显烂的
    survivors = [c for c in candidates if cheap_check(c)]
    # 第2层(中等): Beam Search 估难度,淘汰难度严重偏离目标的
    survivors = [c for c in survivors if beam_difficulty_ok(c)]
    # 第3层(昂贵): 只对幸存者跑完整 SOLVER_PANEL 投票
    return [evaluate_with_solvers(c) for c in survivors]
# 3. 模拟次数 T 的权衡: 随机方差大→T 要够(投票稳定); 但 T 越大越慢。
#    建议: 粗筛阶段 T 小, 终评阶段 T 大。
# 4. 并行: 候选之间、求解器之间、多次模拟之间,全可并行。
```

## 7. 与 Meta 的接口（裸 Core 为基准，Meta 影响为上层叠加）

```
# 原则(呼应第 0 节): 上面整套(生成/求解/评估/调度)全在"裸 Core"上做,不含 Meta。
# Meta(角色技能/铭文)对关卡的影响,是"上层叠加的一层分析",可选、后置:

# 7.1 带 Meta 的画像(平衡性验证)
#   把"带某角色技能的玩家"也抽象成一个 solver 变体:
SOLVER_WITH_META = MCTS(extra_action = character_skill)   # 如: 多一个"借贷"动作(一局一次)
#   用它跑关卡 → 看"带这个技能后,难度下降多少/哪些流派变得过强"
#   → 自动检测角色平衡(呼应早期"用画像验证流派平衡"):
function check_meta_balance(level):
    base = evaluate_with_solvers(level)                    # 裸 Core 基准
    for skill in ALL_CHARACTER_SKILLS:
        with_skill = evaluate_with_solvers(level, panel=panel_with(skill))
        report[skill] = difficulty_drop(base, with_skill)  # 该技能让这关简单了多少
    # 若某技能在所有关卡都大幅降难 → 太强,需削(呼应"难度不被填平")
    return report

# 7.2 接口约定(见 05): 算法引擎对 Core 提供
#   generateLevel(difficulty, profile) -> Level         # 从库里挑/生成
#   validateSolvable(level) -> bool                      # 4.2 的 solvable
#   computeOptimal(level) -> exp_best_score              # MCTS(long) 的期望最优
#   scoreDifficulty(level) -> difficulty                 # 4.3
```

> **关键纪律**: 生成和评估关卡时**默认裸 Core**;Meta 影响单独用「带 Meta 的 solver 变体」分析,只用于**平衡性检测**,不混进关卡质量的基准评估。这样关卡库的难度标签是"纯关卡难度",干净、稳定、不随 Meta 增删而失效。

## 9. 关键纪律清单（实现时反复核对）

1. **裸 Core 为基准**:生成/评估默认无 Meta;Meta 只在平衡分析里叠加(第 0、7 节)。
2. **随机补充→期望思维**:一切评估多次模拟取分布,主力 MCTS,A* 仅小规模校准(第 0、3 节)。
3. **求解器既是玩家也是评估器**:同一套 panel 用于投票定难度 + 生成时筛选(第 2、4 节)。
4. **贪心要像真人**(不能太蠢),它定义地板;MCTS 是近似最优,需 A* 校准(第 3、4 节)。
5. **差距/投票是必要非充分**:差距大保证有深度,但最优解要"可被发现"(非反直觉);好玩的最后一公里仍需真人测试(第 4 节 + `01` 陷阱)。
6. **障碍严格受限**:只用规则清晰、状态确定的少数障碍(冰块/锁/果酱/底层目标);**绝不引入蔓延/增殖型**(巧克力/魔法搅拌器),否则状态爆炸、求解失效(见 `04` 吸收边界)。
7. **解空间度量要准**:用"能产生消除的有效交换数",非全部合法交换(否则高估,节奏曲线失真)(第 3.2 节)。
8. **全程非黑箱**:生成用构造/搜索/约束求解,求解用 MCTS/启发,玩家类型用启发权重——**不引入机器学习**(EA 的 cVAE 路线已否决);每个决策可解释、可证明、可调。
