# 一、对游戏的思考

## 1.1 定位与赌注
- **品类**：gem 消除 match-3（交换相邻、3+ 连消、**带重力下落 + 顶部补给**）。
- **hybrid casual，不是纯消消乐**——纯三消天花板低、机会被 King/Playrix 封死。"纯的没有市场机会了"。
- **规模**：奔**上千关**，是持续运营(live ops)。
- **差异化**：CCS 是"工业长跑 + 稀释换皮"；我们要"**精酿短篇的密度 + 工业长跑的规模**"——靠生成系统在规模上维持设计感，**拒绝换皮**。
- **核心矛盾(=项目全部价值)**：**规模 × 设计感天然对立**。解开它就是这个项目的意义。

## 1.2 两条留存线（缺一不可）
1. **关卡设计本身带留存** ← 本次主攻。
2. **萌宠养成带留存** ← 后续单独立项。注意：**不是 RPG 战力养成**（等级/技能树/组队那套被否决）；萌宠在关卡内是"临时的活元素"（例：会偷懒、影响你消除的兔子），养成是另一条线，形态待定。

## 1.3 match-3 的本质（别做成硬核解谜）
**随机 + 爽感 + 轻策略**：
- **随机性是灵魂不是 bug**：题无唯一解；随机性还是**留存缓冲**（卡关时"再试一次也许运气好"）。
- **爽感(juice)**：casual 玩家图的是爽——连锁 cascade、特殊棋子大爆发、绝杀清场。全程烧脑无爽点=失败品。

## 1.4 hybrid 的本质
**LTV = 留存 × 变现**。关卡最终为 LTV 服务。
- "模拟运营人工生成"的真意：生成系统 = **live ops 的内容引擎**（持续供新关、活动关、按数据调难度）。
- 难度墙既是成就记忆点，也是**变现/回访点**。
- 我们的策略：让"关卡系统本身"成为留存引擎，第二条腿是萌宠养成。

## 1.5 gem 三元素 = 一条"控制力"光谱（随机 → 控制）
- **普通宝石** = 可互换的纯颜色载体 → 随机性之源。
- **特殊宝石** = 玩家用技巧从随机里炼出的可控能量 → 随机→控制的桥（线消=精准/炸弹=区域/色弹=颜色/水晶球=更大）。
- **特殊技能** = 棋盘外的确定性力量 → 确定性之源。
- **好玩 = 在随机里用技巧创造控制、把混沌驯化成秩序**。两极(纯随机=老虎机/纯确定=解谜)都不好玩。

# 二、对关卡的思考

## 本质

### 2.1 每关是"一道题"，不是"一个数字"
独特局面，要玩家*理解+应对*，不是模板换数字。**判据**：有一个"无脑乱消"绕不过的题眼，抽掉它难度就塌。

### 2.2 难度地理 / 补给水文（地形 + 障碍 + 棋子分布）★关键突破
由"沙漏关"截图点醒。gem 有**重力**，决定玩法的不是初始帧，是**消除后补给怎么流动**：
- **地形=河道**(宽口=上游补给足；窄口=咽喉；下游=死水)。
- **障碍=堰坝/礁石**(改变补给 + 本身常是目标)；障碍的难 = 硬度 × 它在水文图上的位置。
- **棋子分布=初始水位/下手点**。
- **设计一关 = 画一张水文图，把目标钉在你要它难的水文位置**。死水区=控制力最难到达处。
- 修正了旧 C++ 的"宝石排布"思路——那只看了静态第 0 帧。

### 2.3 一关的骨架是"四位一体的形状"★最高本质
玩家**完形地**一瞬间形成印象，割裂优化难度=格局小。一关骨架是个**形状**，**设计感=这形状同时是四样东西**：
**难度地理(玩法) × 视觉构图(美) × 直觉指引(可读) × 主题表达(风味)**——四维共振于同一形状，而非各自达标。
（沙漏关四样全干：下游死水/对称美/窄腰喊"下面难"/可挂时之沙主题。）

### 2.4 情绪弧（主体维度）
玩家在一关里走一条情绪弧：上手→中段紧张/卡顿→破局顿悟→收尾爽快清场。**好关卡是设计这条弧，不只是设计难度**。

## 方法

### 2.5 设计感三杠杆
**局面**(静态障碍塑地形) + **活变量**(动态障碍/活元素加压力) + **咬合**(特殊糖组合涌现)。高级关卡=三者叠加。

### 2.6 双难度 + 相对化系数（反数字膨胀）
- **认知难度**(要想多久看懂怎么破) vs **执行难度**(看懂了手上做不做得到)。平庸只抠步数(执行)，高级在认知做文章。**设计感≈认知难度**。
- 参数全部相对化：步数=理论最小步数 × **S**；分数=每步密度 **D** × 步数；障碍=覆盖率 **O** + 种类 **K**(≤3)。

### 2.7 题眼矩阵 = 目标类型 × 阻碍方式
- **目标类型(赢什么)**：清除 / 收集 / 运输(需出口) / 凑分(少用)。**新鲜感主要靠换目标类型**。
- **阻碍方式(怎么难)**：远征(目标在死水下游) / 围城(目标被封先破墙) / 隔断(分区各自为战) / 收割(分散靠大范围组合) / 攻坚(多层量大) / 揭示(藏覆盖物下) / 精算(步数极紧) / 钥匙(只认某组合) / 连环(解一个是解下一个前提)。
- 二维交叉=题眼空间。（**竞速型已砍**——要动态障碍动画、烧美术成本。）

### 2.8 教学关
每出一个新道具(新障碍/特殊糖/目标类型)，配**单独一关**就地教：环境压最简、新道具是唯一变量、几乎不会输。是难度曲线上的低谷。**不是用一段(十几关)去教，是单关、就地。**

### 2.9 六维全景（让设计全面有依据）
关卡不是孤立客体，活在六维里，每维都要被回答：
**① 客体**(玩法/难度/障碍) · **② 主体**(玩家类型/技能成长/情绪弧/认知负荷) · **③ 时间**(单关 1-3min/会话节奏/留存时间结构/内容生命周期) · **④ 感官**(主题/配色/视觉层次/**动画成本**/juice) · **⑤ 经济**(道具/生命/卡关变现/公平 vs 逼氪) · **⑥ 数据**(通关率/流失/转化/求解器/A-B/再平衡)。
我们之前只深耕了①。

### 2.10 settings（一个关卡的完整可调维度）
棋盘(形状/尺寸/孔洞/出口/重力) · 宝石(颜色数/分布/生成器) · 障碍(种类/位置/层数/与目标&特殊糖的交互) · 目标(类型/量/位置) · 资源(步数/技能道具配额/预置特殊糖) · 难度元数据(序列角色/复杂度级/目标通关率/已解锁机制) · 美学(主题/配色/视觉焦点) · 时间(预期单关时长)。

## CCS 研究精华（自己用 API + 搜索挖的，非二手综述）

### 2.11 关卡类型（题眼主轴：靠换目标制造新鲜）
凑分(Target Score，最浅) / 清果冻(Jelly，主菜) / 运原料(Ingredients，路径规划、设计感最高) / 订单收集(Order，可定向逼你造特定控制力) / 限时(已淘汰)。

### 2.12 特殊糖果 + 组合（深度发动机）
- 生成：条纹(恰好4连，清行/列) / 包装(T-L 形 5连，3×3 炸两次) / 色弹(5连直线，清同色) / 果冻鱼(2×2) / 椰子轮。
- **组合矩阵**：条+条=十字；条+包=3行3列大十字；条+色弹=该色全变条纹齐爆；包+包=超大爆；包+色弹=该色全变包装连环爆；**色弹+色弹=清屏**；色弹+普通=清该色。
- **真正的深度在组合(控制力的乘法)，不在障碍数量**。我们的线/炸/色弹/水晶球，要先把两两组合做扎实。

### 2.13 障碍(blocker)真相
看着 30+ 种，机制其实就几个，剩下是皮肤/参数变体。静态三类(覆盖/箱子/锁包)本质同源(占位、需用消除处理掉)。动态类少而精(扩散/生成/倒计时/运输/移动)。**gem 硬约束：好障碍不引入新操作，只给"交换+消除"附加意义。**

### 2.14 King 设计师亲述的七条原理（当方法的尺子）
1. **复杂度阶梯**：难度=复杂度逐级累加，不是拔数值。
2. **交互 > 阻力**：现代障碍要和目标/特殊糖/布局**交互**逼出规划，不是变硬。
3. **障碍要"赚得席位"**：必须有独特交互，否则不加（反换皮硬标准）。
4. **视觉=机制对齐**：长什么样=怎么用，玩家直觉懂。
5. **难度锯齿**：难易交替，难关后给喘息。
6. **资源思维+cascade**：留出"造特殊糖、设级联让关卡自己玩"的空间。
7. **AI bot 估难度 + 人定手感**：机器跑量，人把关 feel。
来源：PocketGamer「复杂度阶梯」、设计师 Rasmus Eriksson 访谈、CCS fandom。

# 三、对实现的思考

## 3.1 做一关的流程（十步）
A 定坐标(① 读序列角色 ② 领复杂度预算) → B 出题(③ 选题眼 ④ 选地形原型 ⑤ 钉目标+布障碍 ⑥ 埋破局路径) → C 调校(⑦ 标难度系数 ⑧ 四重对齐) → D 验证(⑨〔机〕求解器 ⑩〔人〕试玩定手感)。

## 3.2 A→B 的传动轴 = "出题配方"
坐标不直接出题，先翻译成一张**配方**(机制额度/交互深度/目标档/地形档/松紧 S)；B 在配方约束内选。配方=复杂度阶梯的操作化。角色→配方有固定映射(教学/变奏/施压/波峰/喘息各一套)。

## 3.3 约束传播网 + 反向回退（流程的灵魂——不是随便搭）
每步选择被上游**圈定候选子集**、再向下游传约束，自由度越往后越小：
```
坐标 ──过滤──▶ 题眼候选子集
题眼 ──锁定──▶ 地形类型(远征→沙漏 运输→带出口 围城→孤岛 隔断→分叉)
地形 ──锁定──▶ 目标&障碍的位置语法(沙漏→目标钉下游死水)
题眼+地形 ──逼出──▶ 破局路径(不是设计的，是被逼出来的)
破局路径 ──反推──▶ 棋子分布(颜色/空间) + 步数
```
**横向约束**：复杂度预算=全局红线(任一步超支→回退)；求解器=终检(难度不匹配→反向迭代)。
**实质**：流程难在"约束怎么一层层传、错了怎么弹回"，不在步骤顺序。

## 3.4 整体方案：从零到 1000 关
先建「5 库 + 1 求解器」→ 排 1000 关难度曲线 → 每关走同一流水线产出、机器验难度、人工抽检 → 上线数据再平衡。
- **Phase 0 建 5 库**(人主导)：题眼矩阵 · 地形原型库(~8，参数化+四位一体标签) · 障碍集 · 配方表 · 特殊糖组合表。
- **Phase 1 求解器**：输入关卡→输出通关率/步数/解法多样性/难度估计。
- **Phase 2 手搓 20 关**跑通流水线、打磨库与配方。
- **Phase 3 排 1000 关曲线**：每关角色/复杂度级/已解锁机制/主题组；锯齿+复杂度阶梯+机制引入点(配教学关)+主题换皮。
- **Phase 4 批量产出+验证+运营**。
- **人机分工**：人=建库/出题创意/品味抽检；机=实例化/求解器/批量变奏。

## 3.5 引擎现实勘察 ★本次重大发现：能力几乎全有
摸了引擎代码，结论是**我们要的东西 core 层早就做好了**：
- **`core/board.gd` 的构造参数 = 一关的完整 schema**，已支持：
  - `wall_mask` 异形掩码 = **地形/孔洞**(沙漏、孤岛全靠它)
  - `coat`(冰)/`jelly`(果冻) = **覆盖型障碍/清除目标**
  - `choco`(巧克力) = **扩散型动态障碍**(零啃食则蔓延)
  - `cannon`(糖果炮) = **生成型动态障碍**
  - `bomb`(倒计时) = **倒计时型动态障碍**(归零即败)
  - `ing` + `exit_cols` = **运输目标 + 出口**(设计感最高那个)
  - `objectives` = **目标类型**(SCORE/COLLECT/CLEAR_JELLY/拆弹/收原料…)
  - popcorn/cake/mystery = 额外机制(揭示型等)
- **9 种障碍 overlay** 在 `match3/overlays/`(jelly/coat/choco/cannon/bomb/ing/popcorn/cake/mystery)。
- **渲染**(board_view + art/gems + assets/obstacles)、**截图**(`tools/shot_*.gd` 的 get_image→save_png)、**求解器雏形**(`tools/autoplay.gd`，注释自称"贪心求解器雏形") 全有现成范例。
- **数据流**：`levels.json` → `core/level_library.gd`(JSON→Board 工厂) → `Board.new(...)` → `match3/level.gd` 渲染。
- **结论**：那份 levels.json **内容是废的，但 schema 和引擎能力是完整的、好的**。我们**不造引擎，只在它上面包一层薄壳**。实现难度比想象低一个量级。

## 3.6 工具链：AI 可见可编辑的关卡编辑器（第一步）
**核心判断**：第一步不是 GUI。GUI 是给人手(鼠标)的；AI 的"手"是读写文本、"眼"是读文本+读图。所以**让 AI 能编辑的编辑器，地基是"人和 AI 共用的文本关卡格式 + 一键渲染"**，GUI 是后面套上的壳。
```
.lvl (ASCII 源, 人/AI 读写)
  │ [lvl_compile.gd] ASCII 网格 → board.gd 的 grid + 各 layer 数组
  ▼
现有引擎 (core/board.gd + level.gd + board_view.gd) ← 不动它，纯复用
  ├─ [render_lvl.gd] 装配 Board→渲染→get_image→PNG  (抄 shot_gem_burst.gd)
  ├─ [ascii_view.gd] 打印带行列号+图例的 ASCII        (AI 秒读布局)
  └─ [solve_lvl.gd]  --headless 跑 N 次 autoplay→通关率 (抄 autoplay.gd)
```
- **`.lvl` 格式** = board.gd 参数的人类可读版：头部 settings + ASCII board。token 映射：`.`孔洞 `o`随机宝石 `1-6`指定色 `I`冰(coat) `J`果冻(jelly) `C`巧克力(choco) `N`糖果炮(cannon) `B3`炸弹(bomb) `G`原料+出口。
- **AI 工作闭环**：写/改 .lvl → render(出 PNG，Read 看美/直觉) + ascii(核对布局) + solve(看难度) → 判四位一体 → 再改。人同样地用。
- **一套底层 + 两个壳**：编辑器壳(给人)、CLI 壳(给 AI)，共享 加载/渲染/求解 底层。

## 3.7 距离实现还差什么
- **A. 设计资产为零**：5 库现在全是概念，没一个做成可用条目。(难度中)
- **B. 工程现实**：①引擎能力已勘察(几乎全有，好消息)；②关卡 schema 已存在(board.gd 参数)，但需定 .lvl 源格式 + 编译器；③**求解器=最硬骨头**(从 autoplay 雏形升级成蒙特卡洛估通关率)；④编辑器工具。
- **C. 方法从未验证(最大风险)**：整套是推演的，**还没用它真做出一关并证明好玩**。Phase 2 手搓验证要尽早做以打掉此风险。

# 四、对接下来工作的思考

## 4.1 还没确定的关卡设计细节（用户明确指出"很多细节没定"）
- 5 库的**具体内容**：题眼矩阵填格、地形原型库(每个原型的参数/标签)、障碍集到引擎层的最终映射与命名、配方表的精确数值。
- **难度曲线的具体编排**(1000 关序列：机制引入点、教学关位置、主题分组、锯齿)。
- **主题母题库** + 配色/点缀规范 + 动画成本边界。
- **特殊技能/道具系统**的具体设计。
- **变现设计**(道具/生命/卡关点)与关卡的耦合。
- **萌宠养成**(第二留存线)的形态。
- `.lvl` 格式的最终字段细节。

## 4.2 实现优先级（建议）
1. **工具链先行**(让 AI 和你都"能看见")：定 `.lvl` 格式 → 写 `lvl_compile`+`render_lvl`+`ascii_view` → 渲出第一张图。前置：读 `core/level_library.gd` 确认编译目标。
2. **手搓几关端到端跑通**(打掉"方法未验证"风险)。
3. **求解器**(autoplay → 蒙特卡洛估通关率)。
4. **建 5 库** + 排难度曲线 → 规模化。

## 4.3 风险与原则
- 最大风险=方法未验证 → 尽早手搓真关卡验证。
- 工具链命根子=**"出图 + 出数"两个 headless 能力**。
- 不被旧 levels.json / 旧 overlay 概念 / 旧 C++"宝石排布"带偏。
