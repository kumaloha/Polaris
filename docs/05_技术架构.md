# 05 技术架构（Meta / Core 分离）

> 本文档定义工程层面的要求。**核心约束：Meta 与 Core 在代码上必须分离,通过明确接口通信。**
> 这是产品层「Meta 可复用 / Core 可插拔」（见 00、01、02）的技术落地。
> 给技术负责人 / 架构。

---

## 一、分层总则

```
┌──────────────────────────────────────────┐
│  META 模块（独立,可复用）                    │
│  铭文 / 角色 / 抽卡 / 皮肤 / 排行榜 / 变现     │
│  ——不依赖任何具体 Core 的玩法实现             │
└───────────────────┬──────────────────────┘
                    │  通过接口通信（见三）
        ┌───────────┴───────────┐
┌───────▼────────┐    ┌──────────▼───────┐
│ CORE 模块：合成  │    │ CORE 模块：消除   │
│ （可插拔）       │    │ （可插拔）        │
└───────┬────────┘    └──────────┬───────┘
        │                        │
┌───────▼────────────────────────▼───────┐
│  算法引擎（独立服务/模块）                  │
│  关卡生成 / 解的验证 / 难度量化 / 个性化     │
│  ——对每个 Core 提供能力,Core 适配即可       │
└─────────────────────────────────────────┘
```

**三个独立模块**：Meta、Core（每个玩法一个）、算法引擎。三者之间只通过定义好的接口通信,互不直接依赖对方内部实现。

---

## 一·五、实现技术栈与数据契约 ★技术选型★

### 选型
- **游戏端:Godot(GDScript)** —— 表现、动画、输入、消除的运行时玩法、Meta 界面、核心循环。
- **算法端:C++(独立命令行程序)** —— `09` 的全套:关卡生成 + 求解器组投票 + 评估 + 打标签。
- **二者不在代码层耦合**,只通过**「关卡数据文件」**这一个契约连接。

### 为什么这样分（关键决策）
- **速度**:关卡生成 + 求解(尤其 MCTS 多次模拟、大批候选筛选)是计算密集型,GDScript 跑太慢,**必须 C++**。
- **不接 GDExtension,而是用「独立命令行 + 数据文件」**:
  - C++ 做成独立程序,**离线**跑生成/求解/评估,导出关卡库(数据文件);Godot 运行时只**读数据文件、摆盘、玩**,不跑重算法。
  - 好处:① C++ 端**零束缚**——随便用 STL、Google OR-Tools(约束求解 CP-SAT)、SAT 求解器等任何 C++ 库,不受 GDExtension 的 C++ 规范限制 ② 两端**完全独立**,各自迭代、各自调试,互不影响 ③ 算法对错可在 C++ 端单独验证(甚至用 `09` 的求解器投票自测关卡质量),与引擎无关。
  - 这契合 `09` 的设计(关卡库**离线生成**)——数据文件就是关卡库的落地形式。
- **运行时不跑重算法**:Godot 端「从关卡库挑一关来玩」是轻量操作,GDScript 足够。
> 若将来确需**运行时**实时生成/实时求最优(如实时难度调整),再考虑把 C++ 编译成 GDExtension 供 Godot 运行时调用(方式 B);**首发用「独立程序 + 数据文件」(方式 A)即可覆盖,最省事。**

### 数据流
```
┌────────────────────────┐     导出      ┌──────────────────┐     读取     ┌────────────────────┐
│  C++ 算法引擎(离线)      │ ──────────▶ │  关卡数据文件     │ ──────────▶ │  Godot 游戏(运行时) │
│  生成/求解/评估/打标签   │  关卡库      │ (JSON / 二进制)   │  摆盘/玩/表现 │  GDScript           │
│  (见 09)                │              │ = 序列化的 Level  │              │                    │
└────────────────────────┘             └──────────────────┘             └────────────────────┘
        唯一契约 = 关卡数据文件的格式(schema),两端各自独立实现
```

### 关卡数据文件 schema（唯一契约，对应 09 的 Level 结构 + 标签）
```jsonc
{
  // —— 关卡本体(C++ 生成,Godot 据此摆盘) ——
  "level_id": "...",
  "grid": [[ cell, ... ], ...],     // 每格: {type:"EMPTY|TILE|SPECIAL|BLOCKER|WALL",
                                    //        species?:int, special?:"LINE_H|LINE_V|BOMB|COLORBOMB",
                                    //        blocker?:"ICE|LOCK|JAM|JELLY", hp?:int}
  "species_set": [int, ...],        // 本关用哪几种水果(从库 20+ 选 4-6)
  "regions": [...],                 // 战场切割: 区域/隔断定义
  "objectives": [                   // 通关目标(可 MIXED)
    {"type":"SCORE|COLLECT|CLEAR_BLOCKER|BOTTOM", "target":int, "params":{...}}
  ],
  "move_limit": int,                // 步数上限
  "refill_distribution": { "species_weights":[...] },  // 补充棋子随机分布(行业=随机补充)
  "is_scrolling": bool,             // 是否滚动/挖矿形态

  // —— C++ 算出的标签(Godot 据此定难度/推送/竞技) ——
  "difficulty": "EASY|MEDIUM|HARD|EXPERT",
  "lfhc_gap": float,                // 地板低天花板高的程度
  "solspace_curve": [int, ...],     // 局内节奏曲线(解空间随每步变化)
  "vote_distribution": {...},       // 求解器组投票(各类玩家通过率/期望分),可选导出
  "tags": ["region_count:3", "objective:jelly", ...]
}
```
> Godot 端只需实现:**读这个 schema → 摆盘 → 跑消除逻辑(GDScript)→ 用 difficulty/标签 做难度推送和同题竞技**。它不关心关卡怎么生成的(那是 C++ 的事)。

## 二、为什么必须分离（工程理由）

1. **可复用**：Meta 做一次,接多个 Core。若 Meta 和 Core 代码纠缠,换 Core 要重写 Meta,复用红利归零。
2. **可独立迭代**：Meta 改版（加新角色/皮肤/活动）不应动 Core 代码;Core 调玩法不应动 Meta。
3. **可独立测试**：Core 玩法可单独跑（不依赖 Meta 也能玩,守 low floor 的「无角色也能玩」正好也是工程上的解耦验证）。
4. **算法独立**：关卡生成/难度量化是独立能力,对合成、消除都服务,不绑死在某个 Core 里。

## 三、接口约定（关键）

### 3.1 Core → Meta（玩法产出标准化结果）

每局 Core 结束（或过程中）,向 Meta 输出**标准化数据**,Meta 不关心玩法细节：

```
GameResult {
  score: number          // 本局分数（用于抽卡解锁、排行榜）
  resources: {...}        // 产出的资源（用于铭文树等）
  levelId / seed: string  // 本局关卡标识（用于同题竞技比对）
  performanceMetrics: {...} // 难度/表现数据（步数、最优解达成度等,供算法用）
}
```

> 关键：无论 Core 是合成还是消除,都输出同样结构的 `GameResult`。Meta 只读这个结构。

### 3.2 Meta → Core（注入能力与加成）

每局开始前,Meta 向 Core 注入**本局生效的配置**：

```
GameConfig {
  activeCharacter: CharacterAbility   // 当前携带的角色能力（一个）
  runeBonuses: RuneBonus[]            // 当前生效的铭文加成（一组）
  levelData: LevelData                // 算法生成的关卡数据
  skin: SkinConfig                    // 外观（纯视觉,不影响逻辑）
}
```

> Core 负责「让这些配置在本局生效」,但**不负责定义/管理它们**。角色能力、铭文的定义和归属在 Meta。

### 3.3 角色能力接口（重点,保证能力通用）

角色能力要能作用于不同 Core,需定义为「对盘面的标准操作」,而非「对某玩法的特定操作」。建议把能力抽象成通用动作：

- `peek(n)` — 预知未来 n 个单位（主要用于合成;**消除因补充随机、一次掉一批,不做「预知未来棋子」**,见 `02`/`09`）
- `undo(n)` — 撤回 n 步（合成=悔合,消除=悔消）
- `wildcard()` — 通配一次
- `relocate()` — 移动一个单位
- `borrow()` — 借贷（抵押低级换高级,记债）
- `checkDeadlock()` — 死局预警（返回风险,不强制结束）

> 每个 Core 实现这些通用动作在自己玩法里的具体表现。能力定义在 Meta,表现在 Core。**这样新增一个 Core,只要实现这套通用动作,就能复用所有角色。**

### 3.4 算法引擎接口

算法引擎对每个 Core 提供：

```
generateLevel(coreType, targetDifficulty, playerProfile) → LevelData
validateSolvable(coreType, levelData) → boolean       // 保 low floor,不死局
computeOptimal(coreType, levelData) → OptimalSolution  // 定义 high ceiling
scoreDifficulty(coreType, levelData) → difficultyValue // 个性化/分级
```

> Core 提供「该玩法的规则适配器」给算法引擎,引擎据此对该 Core 做生成/验证/求解。换 Core 只需新写一个适配器,引擎主体不变。

---

## 四、Core 可独立运行（重要原则）

**每个 Core 必须能在「无 Meta」状态下正常运行和游玩。**

- 工程上：Core 不硬依赖 Meta;Meta 注入是「增强」,不是「必需」。
- 产品上：这正好对应「没带任何角色/铭文也能正常玩」（守 low floor 的生死线）。
- 好处：Core 可单独开发、单独测试玩法手感,不被 Meta 阻塞。

---

## 五、开发顺序建议（消除主线,对齐 `00` §6 / `09` §8）

1. **先做消除 Core,可独立运行的最小玩法**（不接 Meta,GDScript 实现交换/3连/下落/随机补充/连锁/特效/障碍,验证「光玩消除好不好玩」）。
2. **做 C++ 算法引擎对消除的支持**（按 `09`:生成 + 求解器组投票评估 + 打标签,**离线**导出关卡库;Godot 读库摆盘,见本文一·五数据契约）。
3. **做 Meta 最小版**（一个角色=借贷 + 该角色的铭文 + 分数→抽卡/碎片循环）,通过接口接上消除 Core。
4. **跑通核心循环,真人测试。**
5. （主线验证成功后）**做合成 Core（备选第二 Core,见 `03`）,复用 Meta 和算法引擎**——只新写合成的玩法实现 + 算法适配器 + 通用动作的合成表现。验证复用是否如设计般顺畅。

---

## 六、给技术执行者的提醒

1. **接口先行**：先把 3.1–3.4 的接口定清楚,再各自实现。接口是 Meta/Core/算法三方解耦的契约,不能含糊。
2. **角色能力务必抽象成通用动作**（3.3）——这是「一套角色复用到多 Core」的技术关键。若把能力写死在合成里,换消除时角色全要重写,复用失败。
3. **皮肤纯视觉,不进逻辑层**——皮肤只改渲染,绝不参与任何玩法/数值计算（对应「皮肤不带强度」的产品红线）。
4. **算法引擎做成独立模块/服务**,通过适配器对接 Core,不要把求解逻辑塞进 Core 内部。
5. **客户端存储注意**：若做 H5/小程序原型,避免依赖不被支持的存储 API;持久化走后端或规定的存储方案。
