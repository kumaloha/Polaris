# 总策划文档（Master Design Doc）

> 本文档是整个项目的纲领与路由入口。它定义项目是什么、核心原则、文档结构与阅读顺序。
> **所有执行者（人或 AI）请先读完本文档,再按路由进入对应子文档。**

---

## 0. 一句话定义

一款 **low floor / high ceiling** 的休闲手游：
- **Core（小游戏）**：魔法道具主题的「消除」(吸收开心消消乐/Candy Crush 机制)——人人秒上手(low floor)。角色是独立的二次元人物。
- **Meta（元系统）**：铭文树（量变）+ 角色能力（质变）+ 抽卡（靠高分解锁）+ 皮肤（纯外观）——承接深度、留存与变现（high ceiling）。
- **底层支撑**：算法化关卡生成 + 个性化难度（核心护城河）。

**关键架构思想：Meta 与 Core 解耦。** Meta 是一套可复用框架,Core 是可插拔的小游戏。同一套 Meta 未来可驱动不同 Core（合成、消除等）。

---

## 1. 核心设计原则（不可违背）

这五条是所有设计决策的最高准则,任何子文档的设计若与此冲突,以此为准：

1. **Low Floor, High Ceiling**：大众随便玩能上手能过（地板低）；高手能钻研、能玩出高分、能被看见（天花板高）。一个游戏同时服务两端,不二选一。

2. **不信廉价成长**：任何「变强/成长」都必须付出真实代价。拒绝「投入即变强、无取舍」的数值养成。成长 = 有代价的选择。

3. **难度不被填平**：任何 Meta 能力（铭文/角色）都不能让「做同一件事变简单」从而消解难度；只能「改变玩法、打开新策略」。判断标准：装上后玩家是「更轻松」（砍）还是「要重新思考怎么玩」（留）。

4. **不卖强度,只卖外观与机会**：变现靠广告 + 抽卡（靠实力/高分解锁）+ 皮肤（纯外观）。绝不卖数值强度,绝不破坏公平。

5. **零教育成本 + 不影响投放**：消除元素用全人类共识、零赌博联想、零侵权的符号(魔法道具);角色是独立的二次元人物。确保上手无门槛、分级不升、买量素材无限制。

---

## 2. 三大玩家需求（设计要喂的东西）

所有系统最终服务这三个需求（按重要性）：

1. **确定性的成长路径**——现实成长无法琢磨,游戏提供「努力必兑现」的确定进步。
2. **难度匹配（心流）**——难度贴合玩家水平,需个性化（算法实现）。
3. **To be somebody**——被看见、被认可、与众不同（排行榜、稀有皮肤、高分炫耀）。

> 注：另有「认知卸载/放松」需求,本项目主要服务前三个（唤醒型/进取型）。

## 4.5 实现技术栈（见 `05`）

- **游戏端:Godot(GDScript)** —— 表现、玩法运行时、Meta、核心循环。
- **算法端:C++(独立命令行)** —— 关卡生成/求解/评估/打标签(见 `09`),**离线**产出关卡库。
- **二者不耦合**,只通过**「关卡数据文件」**连接:C++ 导出关卡库 → Godot 读取摆盘开玩。运行时不跑重算法。
- 选 C++ 因生成/求解(MCTS 等)计算密集,GDScript 太慢;用「独立程序+数据文件」而非 GDExtension,使 C++ 零束缚(可用 OR-Tools 等)、两端独立、易调试。

## 5. 给执行者的关键提醒

1. **先做消除 Core（04，主线）+ Meta（02）的最小可玩版本,验证好不好玩。** 合成（03）是可插拔的备选第二 Core,后置——框架可复用是「主线验证成功后」的红利,不是第一步的捷径。

2. **Core 的手感/爽点不可偷懒。** Meta 可复用,但每个 Core 的玩法乐趣是独立的,必须各自打磨到位。框架救不了烂 Core。

3. **算法是护城河,先用 C++ 把关卡生成跑起来。** 按 `09`(伪代码级)实现 C++ 关卡生成器(生成→求解器组投票评估→打标签→导出关卡库),Godot 读关卡库开玩(见 `05`)。这是项目最核心、最不可替代的部分。

4. **守住第 1 节的五条原则。** 尤其原则 3（难度不被填平）——这是大多数同类游戏做废的地方;尤其注意「角色一局一次」「看广告续用有上限」是难度基石。

5. **算法藏在底层。** 大众感觉不到算法,只觉得「这游戏怎么这么舒服/这么懂我」。不要把「优化/最优解」摆到大众面前当门槛。差异化靠算法的「活关卡」,不靠美术(魔法/奇幻消除市场也有,差异化靠算法的「活关卡」而非美术,见 `01`/`06`)。
